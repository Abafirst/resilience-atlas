## Overview

The IATLAS Practice tier provides multi-practitioner group management for clinics and therapy practices. It enables a practice owner to invite colleagues, manage roles, and share access to clinical tools and analytics â€” all under a single Stripe subscription.

**Key capabilities:**
- Multi-seat subscriptions (Practice-5, Practice-10, Practice-25)
- Role-based access control (Owner, Admin, Clinician, Observer)
- Invite-based onboarding with email verification
- Seat utilization tracking and enforcement
- Stripe-backed recurring billing with plan upgrades/downgrades
- Team analytics dashboard

### Middleware

**`requirePracticeRole(roles)`** â€” Express middleware that:
1. Extracts the Auth0 JWT from the `Authorization` header
2. Looks up the user's `Practitioner` record for the requested practice
3. Checks that the user's role is in the `roles` array
4. Returns 403 if the user does not have the required role

**`requireActiveSubscription('practice')`** â€” Client-side utility (`utils/iatlasGating.js`) that:
1. Reads the cached tier from `localStorage`
2. Falls back to a fresh `/api/iatlas/subscription-status` fetch if cache is stale
3. Returns `{ allowed, currentTier, requiredTier }`

### Webhooks

The webhook handler at `POST /api/iatlas/webhook` processes:

- **`customer.subscription.created`** â€” Sets `IATLASSubscription.status = active`, stores `plan` and `practiceId` from metadata, updates `Practice.billing`.
- **`customer.subscription.updated`** â€” Syncs `status`, `currentPeriodEnd`, and `plan` changes.
- **`customer.subscription.deleted`** â€” Sets `status = canceled`, restricts access.
- **`invoice.payment_failed`** â€” Sets `status = past_due`.

## Payment Gating Implementation

### Client-Side Gating

`utils/iatlasGating.js` exports `requireActiveSubscription(requiredTier, token)`:

```javascript
// Returns { allowed: boolean, currentTier: string, requiredTier: string }
const check = await requireActiveSubscription('practice', token);
if (!check.allowed) {
  setPaywallInfo({ currentTier: check.currentTier, requiredTier: 'practice' });
  setShowPaywall(true);
}
```

## Seat Management Logic

1. **Seat limit** is set at practice creation based on the selected plan.
2. **`seatsUsed`** is incremented when a practitioner accepts an invitation.
3. **`seatsUsed`** is decremented when a practitioner is removed.
4. Before sending an invitation, the API checks `seatsUsed < seatLimit`.
5. If the practice is at capacity, the invite endpoint returns `409 Conflict`.
6. Upgrading the plan updates `seatLimit` but does not change `seatsUsed`.

## Role Hierarchy & RBAC

```
Owner  > Admin  > Clinician  > Observer
```

- **Owner:** All permissions. Created automatically at practice creation.
- **Admin:** Can invite/remove members, change roles (except Owner). Cannot access billing.
- **Clinician:** Can access client records and clinical tools. Read-only on team.
- **Observer:** Read-only on all practice resources.

The `requirePracticeRole` middleware enforces this server-side. The frontend also hides UI elements based on the user's role stored in state.
